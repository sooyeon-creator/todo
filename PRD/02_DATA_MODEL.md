# 할일+습관 트래커 -- 데이터 모델

> 이 문서는 앱에서 다루는 핵심 데이터의 구조를 정의합니다.
> 개발자가 아니어도 이해할 수 있는 "개념적 ERD"입니다.

---

## 전체 구조

```
[Task] (description 필드 안에 자유 텍스트 + 체크박스 포함)

[Habit] --1:N--> [HabitLog]
```

SubTask 별도 테이블 없음.
Task의 `description` 필드 하나에 텍스트와 체크박스가 함께 저장됩니다.

---

## 에디터 개념 — 상세 캔버스가 어떻게 생겼는가

Task 제목을 클릭하면 열리는 상세 캔버스는 자유롭게 글을 쓰는 에디터입니다.
텍스트를 쓰다가 들여쓰기 + 특정 단축키(예: `[]` 입력)로 체크박스를 추가할 수 있습니다.

```
┌─────────────────────────────────────────┐
│ 프로젝트 보고서 작성                      │  ← Task 제목
│                                         │
│ 오늘까지 해야 할 내용 정리.               │  ← 일반 텍스트
│ 팀장님 피드백 반영 필요.                  │
│                                         │
│   ☐ 개요 작성        [들여쓰기 체크박스]  │
│   ☑ 데이터 정리                          │
│   ☐ 결론 작성                           │
│                                         │
│ 참고 링크는 노션에 있음.                  │  ← 일반 텍스트
│                                         │
│   ☐ 최종 검토                           │  ← 또 다른 체크박스
│                                         │
│ ████████░░░░░░░ 2/4 (50%)              │  ← Progress Bar
└─────────────────────────────────────────┘
```

**Progress Bar 계산**: 에디터 내 전체 체크박스 수 중 체크된 것의 비율
- 체크박스가 모두 체크되는 순간 Task 자체도 자동 완료 처리 (is_done = true)
- 체크박스가 없는 Task는 수동으로 완료 체크

## 엔티티 상세

### Task
"오늘 해야 할 일 하나" — 상세 캔버스에 자유 형식으로 내용을 작성하는 할 일.

| 필드 | 설명 | 예시 | 필수 |
|------|------|------|------|
| id | 고유 식별자 (자동 생성) | uuid-abc-123 | O |
| user_id | 소유자 (로그인 계정) | uuid-user-456 | O |
| title | 할 일 제목 | "프로젝트 보고서 작성" | O |
| description | 상세 내용 (텍스트 + 체크박스 포함, JSON 형태) | `{"type":"doc",...}` | X |
| is_done | 완료 여부 (수동 체크) | true / false | O |
| done_at | 완료된 날짜시간 | 2026-03-20T14:30:00 | X |
| due_date | 마감일 (Phase 2에서 UI 활성화) | 2026-03-25 | X |
| created_at | 생성 날짜 (자동) | 2026-03-20T09:00:00 | O |

> `description` 필드는 **Tiptap 에디터**의 JSON 포맷으로 저장됩니다.
> Progress Bar는 이 JSON에서 체크박스(`taskItem`) 노드를 파싱해서 계산합니다.

---

### Habit
"매일 또는 주기적으로 반복할 습관" — 한 번 등록하면 계속 나타남.

| 필드 | 설명 | 예시 | 필수 |
|------|------|------|------|
| id | 고유 식별자 (자동 생성) | uuid-hab-321 | O |
| user_id | 소유자 | uuid-user-456 | O |
| title | 습관 이름 | "러닝 30분" | O |
| frequency | 반복 주기 | "daily" / "weekly" | O |
| target_days | 주간 반복 시 요일 목록 (0=일,1=월,...6=토) | [1, 3, 5] (월,수,금) | X |
| target_count | 주 N회일 때 목표 횟수 | 3 | X |
| start_date | 습관 시작일 | 2026-03-01 | O |
| is_archived | 보관 여부 (삭제 대신 숨김) | false | O |
| created_at | 생성 날짜 (자동) | 2026-03-01T10:00:00 | O |

---

### HabitLog
"특정 날짜에 습관을 달성했는지 기록" — 스트릭 계산의 원본 데이터.
체크를 잘못했으면 취소(is_completed = false로 변경)할 수 있습니다.

| 필드 | 설명 | 예시 | 필수 |
|------|------|------|------|
| id | 고유 식별자 (자동 생성) | uuid-log-555 | O |
| habit_id | 속한 Habit의 ID (FK) | uuid-hab-321 | O |
| date | 기록 날짜 | 2026-03-20 | O |
| is_completed | 해당 날 달성 여부 (취소 가능) | true / false | O |

> UNIQUE(habit_id, date) 제약 — 같은 날 중복 기록 방지

## 관계 다이어그램

```
┌──────────────────────────────┐
│            Task              │
├──────────────────────────────┤
│ id (PK)                      │
│ user_id                      │
│ title                        │
│ description (JSONB)          │  ← 텍스트 + 체크박스 전부 여기에
│   └─ taskItem(checked=true)  │     Progress Bar는 이걸 파싱
│   └─ taskItem(checked=false) │
│ is_done  ← 수동 체크         │
│ done_at?                     │
│ due_date?                    │
│ created_at                   │
└──────────────────────────────┘

┌──────────────────┐       ┌──────────────────────┐
│      Habit       │─1:N──▶│      HabitLog         │
├──────────────────┤       ├──────────────────────┤
│ id (PK)          │       │ id (PK)               │
│ user_id          │       │ habit_id (FK)         │
│ title            │       │ date                  │
│ frequency        │       │ is_completed          │
│ target_days[]?   │       │ UNIQUE(habit_id,date) │
│ target_count?    │       └──────────────────────┘
│ start_date       │
│ is_archived      │
│ created_at       │
└──────────────────┘
```

## Progress Bar 계산 공식

```
에디터 JSON에서 타입이 "taskItem"인 노드를 모두 추출

progress = checked = true인 taskItem 수
           ÷
           전체 taskItem 수

예: 4개 체크박스 중 2개 완료 → 2/4 = 50%
```

체크박스가 하나도 없는 Task는 Progress Bar를 표시하지 않습니다.

---

## 왜 SubTask 테이블을 없앴나

기존 설계(SubTask 별도 테이블)는 단순 체크박스 리스트에 적합합니다.
하지만 "텍스트 사이에 들여쓰기로 체크박스를 자유롭게 삽입"하는 방식은
에디터 자체가 문서 구조를 관리하기 때문에 별도 테이블이 불필요합니다.

- **단순해짐**: Task 1개 저장/불러오기가 DB 쿼리 1번으로 끝남
- **유연함**: 텍스트와 체크박스를 섞어서 원하는 위치에 자유롭게 배치 가능
- **Trade-off**: 체크박스 데이터를 서버에서 직접 쿼리할 수 없음 (파싱 필요)

## 확정된 결정사항

| 항목 | 결정 |
|------|------|
| Task 자동완료 | ✅ 체크박스 전부 체크 시 자동 완료 (체크박스 없으면 수동) |
| 체크박스 단축키 | `/[]` 입력으로 체크박스 삽입 |
| 습관 빈도 | daily + 주 N회 모두 지원 |
| 주 N회 스트릭 계산 | 달성한 날 각각 카운트 (ex. 주 3회 → 3일 스트릭 +3) |
| HabitLog 체크 취소 | ✅ 취소 가능 |
| Task 삭제 | 영구 삭제 (휴지통 없음) |
